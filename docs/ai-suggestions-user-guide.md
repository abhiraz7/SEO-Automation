# AI Suggestions & Fix on Page — User Guide

For VTechys marketing workers. No SEO or dev background required — this page
explains every button, badge, and column in the **Issues & Fixes** part of a
project's dashboard.

Open a project at **`/projects/{id}`**, scroll to **Issues by category**, and
click **▼ Optimize** on any row to open the panel this guide covers.

---

## 1. The Optimize panel — six fields, left to right

Clicking **▼ Optimize** on an issue row expands a panel with six fields.
Each is documented below as its own section:

| # | Field | What it's for |
|---|---|---|
| 1 | **Current** | What's actually on the live page right now |
| 2 | **AI Suggestions** | Claude-generated fix candidates |
| 3 | **Edit Suggestion** | Pick one, tweak it, save your decision |
| 4 | **Rule Validation** | Does your chosen text actually pass the SEO rule? |
| 5 | **Judge & Visibility Impact** | 🔧 Coming soon (not built yet — see section 6) |
| 6 | **Action** | Fix on Page / Preview / Skip |

---

## 2. Field 1 — Current

Shows the real value pulled from the last crawl — never a description of the
issue, always the actual text/data. Depending on the issue type you'll see:

- **Text** (title, meta description, canonical) — the exact string on the page.
- **List** (H1s, H2s) — up to 3 shown, with a "+N more" note if there are more.
- **Key/value pairs** (Open Graph, Twitter card fields) — label + value.
- **Images** — a scrollable list of image filenames with their alt text, or a red **Missing** chip if alt text is empty.
- **Schema** — the structured-data types detected, with a "▸ View raw JSON" expander.
- **Markdown** (page content) — an excerpt with a "Show more" toggle for the full text.

The page's URL is always shown underneath, so you always know exactly which
page you're looking at.

---

## 3. Field 2 — AI Suggestions

Click **✦ Generate AI Suggestion** and Claude writes several fix candidates
for this specific issue, using the page's real content as context (not a
generic template). While it's working you'll see a spinning ⚙️ and "Generating
with Claude…" — this normally takes a few seconds.

Each candidate appears as its own card:

- Numbered **1, 2, 3…** in the order Claude ranked them.
- The top card is marked **Best**.
- A small status badge (see section 5) shows whether that specific suggestion has been accepted, edited, or deployed yet.
- Click any card to select it — it highlights and its text drops into the **Edit Suggestion** box (field 3).

If generation fails (network hiccup, Claude API error), you'll see a plain
"Failed to generate suggestions" message with a **↺ Retry** button — never a
fake/blank result.

---

## 4. Field 3 — Edit Suggestion

This is where you make the call on a suggestion. After selecting a card in
field 2:

1. The text appears in the box — edit it freely if you want to change wording, length, etc.
2. Click **Save**:
   - If you **didn't change the text** → it's recorded as **Accepted** as-is.
   - If you **did change the text** → it's recorded as **Edited**, and your version is saved *alongside* the original (Claude's original suggestion is never overwritten — useful for comparing later).
3. Click **Cancel** to clear the box without saving anything.

A status chip appears next to the buttons confirming what just happened
("accepted" / "edited"), and the matching card in field 2 updates its badge
to match — no page reload needed. If something goes wrong (network error,
already deployed, etc.) you get an error toast, not a silent failure.

> **Rule of thumb:** Save always means "I've made a decision." Nothing is
> ever recorded until you click it.

---

## 5. Status badges — what each color means

You'll see these badges throughout the panel and in the Fix on Page modal:

| Badge | Color | Meaning |
|---|---|---|
| **pending** | Gray | Generated by AI, no decision made yet |
| **accepted** | Green | You approved it as-is |
| **edited** | Blue | You approved a modified version |
| **rejected** | Red | You dismissed it — won't be offered again unless regenerated |
| **deployed** | Purple | Live on the WordPress site right now |

Regenerating suggestions for an issue never deletes accepted/edited/deployed
rows — only undecided ones get replaced. Your decisions are permanent history.

---

## 6. Field 4 — Rule Validation

Tells you whether the text currently in the **Edit Suggestion** box would
actually pass this issue's automated SEO rule (length limits, required
elements, etc.) — a live spot-check before you commit to it.

- **Failed** (red) / **Needs Review** (amber) shown by default, based on the original issue's severity.
- Start typing or select a suggestion card, and it re-checks automatically (on blur) and shows **Passed** ✓ or **Failed** ✕ with the specific reason.
- If there's no automated rule for that field type, it says so plainly ("No automated rule for this field — review manually") instead of guessing.
- **View all rules** opens a reference drawer showing every rule for that issue category.

---

## 7. Field 5 — Judge & Visibility Impact — 🔧 Coming soon

This field is intentionally shown as a single muted **Coming soon** chip. It
used to show fake-looking numbers ("Not Judged", "—/10", per-AI-model
scores) that implied working features — that was misleading, so it's been
collapsed into one honest placeholder. Two features are planned here:

- **LLM Judge** — an automated quality score for each suggestion.
- **AI Visibility Impact** — estimated change in how often AI answer engines (ChatGPT, Gemini, etc.) would cite this page if the suggestion were applied.

Neither exists yet. Nothing to click here today.

---

## 8. Field 6 — Action

Three buttons per issue row:

- **Fix on Page** — opens the Fix on Page modal (section 9) with this exact issue front and center.
- **Preview** — placeholder, not yet wired up.
- **Skip** — closes the panel without doing anything (same as clicking ▲ Close).

---

## 9. Fix on Page modal

Clicking **Fix on Page** opens a popup instead of navigating away, so you
never lose your place in the dashboard. It shows:

- **Current Score** — the page's audit score as a circular gauge, plus a **Re-Audit** button to force a fresh check.
- **WordPress link status** — "🔗 Linked to WordPress post #N" in green if the tool already knows which WordPress post this page maps to, or a gray note that Deploy will ask for it if not (see section 10).
- **Title Tag** — current value + the same pass/fail rule checklist you'd see on the full page view.
- **📍 Selected issue** — the exact issue you clicked "Fix on Page" from, highlighted at the top with its own **Get Suggestions** button and the full accept/reject/edit/deploy flow for each candidate.
- **Other issues on this page (N)** — every other issue on the same page, collapsed under a divider so they don't compete for attention with the one you came to fix. Click to expand if you want to work on something else without leaving the modal.
- A link at the bottom to open the full raw page data (meta tags, headings, images, schema) in a new tab, for the rare case you need the deep technical view.

## 10. 🚀 Deploy to WordPress

Once a suggestion is **Accepted** or **Edited**, and its field type supports
deployment (currently: meta description, SEO title, and H1/post title —
image alt text is not yet supported), a **🚀 Deploy** button appears.

**What happens when you click it:**

1. The tool tries to figure out *which WordPress post* this page is automatically — no typing required, in most cases.
2. It reads the *current* live value from WordPress first (so a rollback later has something real to restore).
3. It writes your accepted/edited text to the site.
4. The suggestion's badge flips to **deployed** (purple), and it's recorded in Revision History (section 11).

**If it can't find the WordPress post automatically** — e.g. this is a
homepage/root URL, or the page hasn't been crawled since the WordPress
connection was set up — you'll get a one-time prompt asking for the numeric
WordPress post ID. This is the exception, not the normal path: once a page
has been crawled *after* a WordPress connection is configured, its post ID
is looked up and remembered automatically, and you won't be asked again for
that page.

Once a suggestion is deployed, it's locked — you can't accept/reject/edit it
further until you roll it back.

---

## 11. ↩ Roll back & Revision History

- Any **deployed** suggestion shows a **↩ Roll back** button instead of the usual Accept/Reject/Edit — clicking it writes the *original* live value back to WordPress and flips the suggestion back to **accepted** (your decision to use it still stands; only the live deploy is undone).
- The **📜 Revision History** drawer (top of the project page) lists every deploy and rollback ever made on this project — what changed, before/after values, when, and whether it's since been rolled back. This is your audit trail if a client asks "what did you actually change on our site."

---

## 12. Quick reference — what to do when

| You want to… | Do this |
|---|---|
| See what's actually on the page right now | Field 1 — **Current** |
| Get AI-written fix ideas | Field 2 — **✦ Generate AI Suggestion** |
| Approve a suggestion as-is | Select a card → **Save** (no edits) |
| Approve your own tweaked version | Select a card → edit the text → **Save** |
| Check if your text will actually pass the rule | Field 4 — **Rule Validation** (updates live as you type) |
| Work on one issue without losing sight of the rest | **Fix on Page** → focused issue at top, others collapsed below |
| Push a fix live to the website | **🚀 Deploy** on an accepted/edited suggestion |
| Undo something you deployed | **↩ Roll back** on that suggestion |
| See everything ever deployed | **📜 Revision History** drawer |
| Check if a page is linked to WordPress | Fix on Page modal → WordPress link status line |

---

*If Deploy keeps asking for a post ID on the same page every time, it means
that page hasn't been re-crawled since the WordPress connection was saved —
run Crawl or Re-Audit on it once and the tool will remember going forward.*
